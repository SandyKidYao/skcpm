# 文档同步更新

根据代码检查并更新项目中过时的文档。核心原则：**代码是唯一事实来源，永远根据代码更新文档，绝不根据文档修改代码。**

## 工作流程

### 第一步：确定目标文档

根据 `$ARGUMENTS` 决定要检查的文档：

**有参数时**：直接使用用户指定的文件路径。

**无参数时**：自动扫描项目，查找以下类型的文档文件：
- 根目录：`README.md`、`CLAUDE.md`、`CONTRIBUTING.md`、`CHANGELOG.md`
- 文档目录：`docs/`、`doc/`、`documentation/` 下的 `.md` 文件
- 部署相关：包含 deploy、setup、install 等关键词的文档
- 接口文档：包含 api、interface、endpoint 等关键词的文档
- AI 开发指导：`CLAUDE.md`、`.cursorrules`、`.github/copilot-instructions.md` 等

使用 Glob 工具扫描，列出找到的文档文件清单，让用户确认要检查哪些文档。如果文档数量较多（>5个），建议用户选择优先检查的文档。

### 第二步：逐份文档分析

对每份目标文档执行以下流程：

1. **阅读文档全文**：使用 Read 工具读取文档内容，理解文档描述的主题和范围。

2. **提取关键断言**：识别文档中对代码行为的具体描述，包括但不限于：
   - 文件路径和目录结构的描述
   - 函数、类、接口的名称和用法
   - 配置项、环境变量的说明
   - 命令行用法和参数说明
   - 架构和数据流的描述
   - 依赖和版本要求
   - 部署步骤和前置条件

3. **对照代码验证**：对每个关键断言，使用 Grep、Glob、Read 工具在代码中验证：
   - 文档提到的文件/目录是否存在
   - 函数签名、参数、返回值是否与文档一致
   - 配置项名称和默认值是否正确
   - 命令和参数是否与实际实现匹配
   - 架构描述是否反映当前代码结构

4. **记录差异**：记录每个不一致之处，格式为：
   - 文档原文（引用）
   - 代码实际情况
   - 差异性质（过时/错误/缺失/多余）

### 第三步：更新文档

对发现的差异执行更新：

1. **展示差异清单**：先向用户展示所有发现的差异，按严重程度排序：
   - **错误**：文档描述与代码完全矛盾
   - **过时**：文档内容不再反映当前代码
   - **缺失**：代码中有重要变更但文档未提及
   - **多余**：文档描述的功能已被删除

2. **执行更新**：使用 Edit 工具逐项修改文档内容，确保：
   - 保持文档原有的结构和风格
   - 保持文档原有的语言（中文文档用中文更新，英文文档用英文更新）
   - 只修改与代码不一致的部分，不改动仍然准确的内容
   - 不添加文档中原本没有的章节或内容（除非是补充缺失的关键信息）
   - 更新后的描述要准确反映代码的实际行为

3. **输出更新报告**：完成后输出简要的更新摘要，列出每个文档的修改项。

## 注意事项

- **绝不修改代码**：即使发现代码可能有 bug，也只更新文档来反映代码的实际行为。如果怀疑是代码 bug，在更新报告中标注提醒用户。
- **保守更新**：拿不准的地方不改，宁可漏改也不要误改。存疑之处在报告中标注，让用户判断。
- **大文档分段处理**：对于超长文档，按章节分段验证，避免遗漏。
- **关注实质差异**：忽略纯格式差异（如空格、换行风格），聚焦内容准确性。
